# Spec 007 — GitHub Actions CI

**Expert:** Hiroshi Nakamura
**Depends on:** Phase 1 (tooling — lint, typecheck, test scripts must exist)
**Can run in parallel with:** Phase 6 (OAuth)

## What and Why

Every PR against `main` must be validated automatically. The pipeline runs lint, typecheck, and tests. It does not deploy. It does not need a database — all tests use mock adapters and in-memory state.

One job. Turborepo handles the parallel task graph internally. No matrix, no separate jobs per package.

## Scope

**In:** `.github/workflows/ci.yml`
**Out:** deployment pipeline (separate workflow when needed), Docker build in CI (added later), database integration tests in CI, PR preview environments

---

## File: `.github/workflows/ci.yml`

```yaml
name: CI

on:
  pull_request:
    branches: [main]
  push:
    branches: [main]

concurrency:
  group: ci-${{ github.ref }}
  cancel-in-progress: true   # cancel stale runs on new pushes to the same branch

jobs:
  ci:
    name: Lint, Typecheck & Test
    runs-on: ubuntu-latest
    timeout-minutes: 10

    steps:
      - name: Checkout
        uses: actions/checkout@v4

      - name: Setup pnpm
        uses: pnpm/action-setup@v4
        with:
          version: 9.15.4   # must match packageManager in root package.json

      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: 22
          cache: pnpm

      - name: Install dependencies
        run: pnpm install --frozen-lockfile

      - name: Typecheck
        run: pnpm typecheck

      - name: Lint
        run: pnpm lint

      - name: Test
        run: pnpm test
        env:
          # Provide test env vars so config.ts validation passes
          DATABASE_URL: postgresql://dojo:dojo@localhost:5432/dojo_test
          SESSION_SECRET: ci-test-secret-minimum-32-characters-long
          GITHUB_CLIENT_ID: ci-test-client-id
          GITHUB_CLIENT_SECRET: ci-test-client-secret
          LLM_BASE_URL: http://localhost:11434
          LLM_API_KEY: ci-test-api-key
          WEB_URL: http://localhost:5173
          NODE_ENV: test
```

## Key Decisions

**Why `concurrency: cancel-in-progress`?**
If you push two commits quickly, the first run becomes irrelevant. Cancel it automatically to save CI minutes and get faster feedback on the latest commit.

**Why no `DATABASE_URL` pointing to a real PostgreSQL?**
CI tests are unit tests and application-layer tests using `MockLLMAdapter` and in-memory stubs. Integration tests (against real PostgreSQL) are run locally with the devcontainer. Adding a `services: postgres:` container to CI is future work, enabled when integration tests are written.

**Why `--frozen-lockfile`?**
If `pnpm-lock.yaml` is out of sync with `package.json`, CI fails immediately. Prevents "works on my machine" dependency drift.

**Why one job?**
Turborepo computes the task dependency graph and runs tasks in parallel internally. Splitting into separate GitHub Actions jobs adds matrix overhead and inter-job artifact passing complexity without benefit at this monorepo size.

**Why `timeout-minutes: 10`?**
Tests with mock adapters should complete in under 2 minutes. 10 minutes is a generous ceiling that catches infinite loops or hung processes without waiting the GitHub default 6 hours.

**Step order: typecheck before lint before test.**
Typecheck catches the broadest errors fastest (type errors surface before lint). Lint catches style and import cycle violations. Tests run last — no point running tests if the code doesn't compile.

## Optional: Turbo Remote Cache

If a Turbo remote cache account is available, add to the `ci` job environment:

```yaml
env:
  TURBO_TOKEN: ${{ secrets.TURBO_TOKEN }}
  TURBO_TEAM: ${{ secrets.TURBO_TEAM }}
```

This caches build artifacts across CI runs and cuts repeat job time significantly. Add the secrets to the GitHub repository settings when available.

## Optional: Coverage Report as PR Comment

Add after the test step (optional, adds complexity):

```yaml
      - name: Upload coverage
        uses: actions/upload-artifact@v4
        if: always()
        with:
          name: coverage-report
          path: apps/api/coverage/
          retention-days: 7
```

## Branch Protection Rules (manual setup in GitHub)

After the workflow is pushed, configure branch protection on `main`:
- Require status checks: `Lint, Typecheck & Test`
- Require branches to be up to date before merging
- Do not allow bypassing the required status checks

These are set in GitHub repository settings, not in code.

## Acceptance Criteria

- [ ] The workflow file exists at `.github/workflows/ci.yml`
- [ ] Opening a PR against `main` triggers the `CI` workflow
- [ ] Pushing to `main` triggers the `CI` workflow
- [ ] A PR with a failing test blocks merge (requires branch protection to be configured)
- [ ] A PR with a lint error (`import/no-cycle` violation) fails the `Lint` step
- [ ] A PR with a TypeScript error fails the `Typecheck` step
- [ ] The workflow completes in under 5 minutes on a clean run

## Out of Scope

- Deploy on merge to `main`
- Docker build in CI (deferred — adds 3-5 minutes to CI time without coverage benefit at Phase 0)
- End-to-end tests in CI (requires database service container + full app — post-Phase 1 concern)
- Dependabot or Renovate for dependency updates (add when the foundation is stable)
