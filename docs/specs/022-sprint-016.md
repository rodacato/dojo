Añadir botón "Surprise me" junto al CTA existente. Al hacer click:

1. Llama al endpoint de ejercicios existente con las preferencias del usuario

3. Navega directamente a `/kata/:sessionId` sin pasar por la pantalla de selección

```tsx
// Nuevo handler en DayStartPage
async function handleSurpriseMe() {
  setSurpriseLoading(true)
  try {
    const options = await api.getExerciseOptions({ mood, duration })
    if (options.length === 0) return
    const picked = options[Math.floor(Math.random() * options.length)]
    const session = await api.startSession({ exerciseId: picked.id, mood, duration })
    navigate(`/kata/${session.id}`)
  } finally {
    setSurpriseLoading(false)
  }
}
```

## Part 2 — Fix-the-bug kata (5 ejercicios)

### 2.1 Schema — sin cambios

Los ejercicios de "fix the bug" usan el schema existente:
- `type: 'code'`
- `category: 'debugging'`
- `starterCode`: código con el bug (ejecuta pero falla los tests)
- `testCode`: suite que pasa con el fix, falla con el código roto
- `variations`: 2 variaciones del prompt del sensei (evalúa causa raíz, no solo el patch)

### 2.2 Prompt del sensei — variaciones

Las variaciones del sensei para ejercicios `category: 'debugging'` deben incluir contexto:

**Variación A (Sensei estricto):**
```
This is a bug-fix exercise. The developer was given broken code with a specific bug.
Evaluate: (1) Did they identify the root cause, not just patch the symptom?
(2) Is their fix minimal and targeted, or did they rewrite unnecessarily?
(3) Did they explain what was wrong?
```

**Variación B (Sensei formativo):**
```
This is a debugging exercise. The goal is root cause identification.
Focus your evaluation on: the developer's debugging approach, whether they understood
WHY the code was wrong (not just THAT it was wrong), and the quality of their fix.
```

Estas instrucciones se añaden al prompt base como contexto adicional cuando `category === 'debugging'`.

## Part 3 — SQL advanced kata (5 ejercicios)

### 3.1 PostgreSQL en Piston — confirmado

ADR 014 ya cubre esto. Los ejercicios SQL existentes ya usan `language: 'sql'` con Piston.

### 3.2 testCode pattern para SQL

El testCode SQL en Piston funciona ejecutando un script que:
1. Crea las tablas con datos de prueba (INSERT)
2. Ejecuta la solución del usuario
3. Valida el resultado con un SELECT de verificación
4. Imprime PASS o FAIL con descripción

```sql
-- Patrón estándar para testCode SQL
CREATE TEMP TABLE employees (id INT, name TEXT, dept TEXT, salary INT);
INSERT INTO employees VALUES
  (1, 'Alice', 'Engineering', 95000),
  (2, 'Bob', 'Engineering', 85000),
  (3, 'Carol', 'Marketing', 75000),
  (4, 'Dave', 'Marketing', 80000);

-- [Solución del usuario se inserta aquí como CTE o query]

-- Verificación
DO $$
DECLARE result_count INT;
BEGIN
  SELECT COUNT(*) INTO result_count
  FROM (/* query del usuario */) r
  WHERE r.rank = 1 AND r.name IN ('Alice', 'Dave');

  IF result_count = 2 THEN
    RAISE NOTICE 'PASS: Department leaders correctly identified';
  ELSE
    RAISE EXCEPTION 'FAIL: Expected 2 top earners, got %', result_count;
  END IF;
END $$;
```
