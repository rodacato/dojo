# UX/UI flow gaps — 2026-04

**Sprint:** 020 Part 1
**Author:** Kira Tanaka (via Soren C6 lens)
**Scope:** `apps/web` — all user-facing flows.

Audit output. Not a fix list — a *map of friction* classified by severity. The sprint's Part 1 ships 3-5 of these in-sprint; the rest get promoted to backlog (Triaged — later) and referenced by issue.

Classification:
- **Blocker** — breaks or stalls a flow the user is actively trying to complete
- **Friction** — flow works but with noticeable UX cost (silent failure, ambiguity, dead-end)
- **Polish** — cosmetic inconsistency or dev-facing smell, no immediate user cost

---

## Blockers

### B-1 · Kata prepare timeout is 60s before error UI

`KataActivePage` polls the `prepare` endpoint up to 30 times (~60s) before showing `prepareError`. During those 60s the screen looks hung — no retry affordance, no progress indicator beyond the initial spinner.

**Impact:** learner thinks the app is broken and abandons. Most common first-run bug we've seen.
**Location:** `apps/web/src/features/kata/KataActivePage` (polling loop in the prepare hook).
**Fix sketch:** reduce ceiling to ~20s, surface a "still preparing…" message after 8s, and show `prepareError` with a Retry + "Back to dashboard" pair as soon as the server returns a non-transient failure.

### B-2 · Results page breaks on reload (sessionStorage dependency)

`/kata/:id/result` reads `dojo-attempt-{sessionId}` from sessionStorage to display the session. If the user reloads, the key was already cleared by the eval screen → the page falls back to a silent null state or error. No "this result can't be loaded, go to history" affordance.

**Impact:** any user who hits reload on their results loses the view of what they just completed.
**Location:** `ResultsPage` + `useEvaluationStream`.
**Fix sketch:** don't depend on sessionStorage for result display — hydrate from the API using `sessionId`. SessionStorage should be cache, not source of truth.

## Friction

### F-1 · Silent error swallowing in critical paths

Several `.catch(() => {})` or `setError(true)` paths with no retry UI:
- `CoursePlayerPage` line ~79 — anonymous→auth progress merge failure silently drops all prior progress (indistinguishable from "no progress made")
- `PublicProfilePage` — 404 renders "Failed to load" with no retry, no link back
- `SharePage` — "doesn't exist or isn't public" is the only error for any failure (network, 500, private, deleted)
- `LandingPage` line ~415 — waitlist/auth error caught and dropped

**Impact:** learner can't distinguish "server problem" from "not for me" — retries blindly or abandons.
**Fix sketch:** consistent error states via a small `<ErrorState />` component: icon, message, retry button, secondary link. Reuse across 4 call sites.

### F-2 · Evaluation auto-redirect is silent (1.5s)

`useEvaluationStream` auto-navigates from `/kata/:id/eval` to `/kata/:id/result` 1.5s after the final evaluation token arrives. No toast, no notification, no "Finished — taking you to results…" message. The page just changes.

**Impact:** disorienting for first-time users; users who glanced away miss that something happened.
**Fix sketch:** show "Evaluation complete — opening results…" above the fold for the 1.5s window; keep the auto-nav.

### F-3 · WebSocket close codes handled inconsistently

Code `4001` redirects to home with `?error=session_expired`. All other close codes fall through to a silent disconnect — the streaming screen freezes with partial output.

**Impact:** user sees partial sensei response, thinks it's thinking, waits indefinitely.
**Location:** `useEvaluationStream` WS handler.
**Fix sketch:** default branch should show "Connection lost — reconnect?" with a retry button.

### F-4 · Settings is fire-and-forget without a save gesture

Every `SettingsPage` field saves on change with a small "saving…/✓ saved" indicator. Users used to explicit Save buttons don't trust that the change took. There's no undo if they toggled by accident.

**Impact:** low-grade anxiety, occasional "did that save?" reload.
**Fix sketch:** either (a) keep fire-and-forget but add a toast on success, or (b) switch to a Save button per section. Panel recommendation: (a) is cheaper and consistent with Stitch.

### F-5 · DayStart "Surprise me" makes a redundant API call

Clicking "Surprise me" fetches even when mood + duration are already selected. Minor wasted call but also means a visible spinner where none is needed.

**Location:** `DayStartPage`.
**Fix sketch:** skip the fetch when state is already complete; surprise-me just resolves to "go".

### F-6 · Anonymous→auth course progress merge is not idempotent

`CoursePlayerPage` runs the merge on every auth state change. If it fires twice (e.g., focus loss + focus gain with a refreshed token), duplicate merge calls can race.

**Impact:** rare, but when it triggers the server returns 409s that get swallowed → progress appears inconsistent between sidebar and dashboard.
**Fix sketch:** gate on a `hasMerged` ref or move the merge to a single-shot effect keyed on userId.

## Polish

### P-1 · Inconsistent page headers

`HistoryPage` renders `<LogoWordmark />`, `ResultsPage` does not. Same AppShell, different headers. Choose one.

### P-2 · Loading states split between PageLoader and ad-hoc

Most pages use `<PageLoader />` but `LearnPage` checks `!courses` and `ResultsPage` has its own loading shape. Pick one contract per layout.

### P-3 · `console.error` in production code

`AuthContext:31`, `ResultsPage:20`, `AdminEditExercisePage:71` — dev-facing noise only, but should go through a logger (or be removed).

### P-4 · Admin refresh() without await on mount

`AdminCoursesPage` line ~38: `refresh().catch(...)` on mount is not awaited, so the initial loading boundary is lossy. Replace with a useEffect that awaits.

### P-5 · Web test coverage is 1 file for 66 pages/components

Not a fix for this sprint, but worth flagging: `slots.test.ts` is the only frontend unit test. Regression risk on UI changes is high. Triage a "testing backbone" sprint later (Hiroshi S1).

### P-6 · SharePage uses raw `fetch` instead of the API client

Inconsistent with the rest of the app; means it bypasses any interceptors/error normalization.
