# Roadmap

Incremental by design. Each phase is usable and valuable on its own before moving to the next. No phase is started until the previous one is being actively used.

---

## Phase 0 — The Empty Dojo
*Goal: One user (the creator) can complete the full kata loop.*

The core loop, nothing else. No social features, no leaderboard, no invitations. Pure dogfooding to validate that the practice itself is valuable before building anything around it.

**Scope:**
- [ ] Monorepo setup (Turborepo, React+Vite, Hono, Postgres)
- [ ] Docker Compose for local dev and production
- [ ] Kamal deploy config for Hetzner VPS
- [ ] GitHub OAuth (single user initially)
- [ ] Exercise and Variation tables with seed data (5-10 hand-crafted kata)
- [ ] Mood + time filter → 3 exercise options screen
- [ ] Session creation: variation selected, body generated by LLM, persisted
- [ ] Split-panel kata view: context + code editor / chat / Drawhaus embed (by type)
- [ ] WebSocket evaluation flow: user submits → sensei streams response → follow-up or verdict
- [ ] Session marked complete or failed
- [ ] Basic post-kata analysis screen (the sensei's final verdict, topics to review)
- [ ] `/health` endpoint, basic logging

**Done when:** The creator uses it daily for at least one week without friction.

---

## Phase 1 — Open the Doors
*Goal: Invite friends. The dojo becomes social.*

- [ ] User table and multi-user support
- [ ] GitHub OAuth for all users
- [ ] Public profile page (username, avatar, streak, recent kata, badges earned)
- [ ] Invitation system (creator sends invite links, limited seats)
- [ ] Share card generation (OG image with verdict, exercise name, memorable sensei quote, GitHub handle)
- [ ] "No repeat within 6 months" rule enforced per user
- [ ] Basic streak tracking (days with at least one completed kata)
- [ ] `FIRST KATA` and `5 STREAK` badges

**Done when:** At least 3 invited users complete kata consistently over 2 weeks.

---

## Phase 2 — The Scoreboard
*Goal: Light gamification that motivates without corrupting the practice.*

- [ ] Leaderboard between invited users (consistency-based, not score-based)
- [ ] Full badge system: `POLYGLOT`, `ARCHITECT`, `BRUTAL TRUTH`, `CONSISTENT`, `UNDEFINED NO MORE`, `SENSEI APPROVED`, `SQL SURVIVOR`, `RUBBER DUCK`
- [ ] Dashboard improvements: weak areas identified by LLM over time, topics to revisit
- [ ] Psychological analysis view: patterns in how you respond (do you rush? avoid certain types? give up early?)
- [ ] Drawhaus diagram saved per session (whiteboard kata history)
- [ ] Exercise feedback loop: after completing a kata, user can rate and comment on a future exercise in planning

**Done when:** Users are checking their dashboard and badges voluntarily, not just completing kata.

---

## Phase 3 — Feed the Dojo
*Goal: Content creation scales beyond the creator.*

- [ ] Invited users can submit exercise proposals
- [ ] LLM QA gate: proposed exercises are evaluated for quality, clarity, and completeness before review
- [ ] Creator review queue: approve, reject, or request changes
- [ ] Exercise versioning: update a kata without breaking active sessions
- [ ] Admin endpoints for exercise management (used by Dojocho when built)
- [ ] `created_by` attribution on exercises: contributors get credit on the exercise card

**Done when:** At least 20% of published kata come from contributors, not the creator.

---

## Phase 4 — Evaluate Opening
*Goal: Decide whether to open to the public or stay curated.*

This phase is a decision, not a build.

Questions to answer at this point:
- Is the quality of exercises good enough for strangers?
- Does the honor code hold at scale or does it need enforcement?
- Is there enough content diversity to serve developers with different stacks?
- Is there a sustainable way to fund the LLM costs if usage grows?

Options:
- **Stay invite-only** — maintain quality and community, grow slowly
- **Open with waitlist** — control growth while building content
- **Full public** — accept the chaos, iterate fast

No decision made until Phase 3 is running well.

---

## Out of Scope (for now)

These ideas exist but won't be built until the core loop is proven:

- **Dojocho** — dedicated curation tool for building kata at scale
- **Team dojos** — shared practice spaces for engineering teams
- **Native mobile** — the web app is sufficient, mobile browser works
- **AI-generated exercises on demand** — too variable in quality for the current content bar
- **Timed competitions** — contradicts the "practice over competition" philosophy
- **Certificates or badges for employers** — not the product's purpose

---

## Principles That Guide Prioritization

1. **Use it before building around it.** No feature gets built until the previous phase is in real daily use.
2. **The loop comes first.** The kata → evaluation → analysis loop is the product. Everything else is context.
3. **Complexity is the enemy.** If a feature requires the user to think about the product instead of the practice, it's probably wrong.
4. **The sensei's honesty is sacred.** No feature that softens, gamifies, or incentivizes the evaluation to be nicer than it should be.
